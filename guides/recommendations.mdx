NQ's recommendation engine is built on a graph database (Neo4j). Every media item, creator, tag, genre, and user is a node, and the edges between them capture relationships — who made something, what it shares with other titles, and what you have consumed. Recommendations are generated by traversing this graph from your activity outward to titles you have not yet logged.

---

## How the engine works

When you add media and log activity, NQ builds a personal preference graph:

1. **Activity nodes** — each item you log creates a `UserActivity` edge between your `User` node and the `Media` node.
2. **Relationship traversal** — the engine walks outward from items you have rated highly or marked as completed, following edges to related media (shared creators, tags, genres, franchises, and subjects).
3. **Fitness scores** — each candidate title receives a `score` on the `Recommendation` type. Higher scores surface to the top of your queue.
4. **Recommendation nodes** — results are stored as `Recommendation` nodes linked to your user, so the app can display them without re-running the graph traversal on every load.

```graphql
type Recommendation {
  id: UUID!
  user: User!
  media: Media!
  recommender: User    # null for engine-generated recommendations
  source: String       # origin of the recommendation
  score: Float         # fitness score; higher is better
}
```

## Viewing recommendations

The **Recommended** tab on the home screen displays your current recommendations as a cover grid. Items are fetched via the `getRecommendations` GraphQL query and ordered by score.

Pull down on the home screen to refresh the list after adding new media or completing a sync.

---

## The `getRecommendations` query

You can query recommendations directly through the GraphQL playground at `http://localhost:8080`.

```graphql
query {
  getRecommendations {
    id
    score
    source
    media {
      id
      title
      ... on Movie {
        releaseDate
        description
        coverUrl
        averageRating
        genres {
          name
        }
      }
      ... on TVShow {
        releaseDate
        description
        coverUrl
        averageRating
      }
      ... on Game {
        releaseDate
        description
        coverUrl
        genre
      }
      ... on MusicAlbum {
        releaseDate
        coverUrl
        label
      }
      ... on Book {
        releaseDate
        coverUrl
        isbn
      }
    }
  }
}
```

The query returns recommendations for the currently authenticated user (`me`). Each item includes the media object with all fields available for that type.

## The `relatedMedia` field

Every media type in the schema exposes a `relatedMedia` field that returns other titles connected to it in the graph. The field is declared on the base `Media` interface, so it is available on all five concrete types — `Movie`, `TVShow`, `Book`, `Game`, and `MusicAlbum`:

```graphql
interface Media {
  relatedMedia(limit: Int): [Media!]!
}
```

The `limit` argument caps how many related items are returned. Omit it to receive all connected titles.

This field is populated asynchronously after a media entry is first added. The backend runs a recursive search to discover titles connected by shared creators, tags, genres, or franchises — using TMDB for films and TV, IGDB for games, and Open Library for books. You can query it from any item:

```graphql
query {
  media(id: "some-uuid") {
    title
    relatedMedia(limit: 10) {
      id
      title
      ... on Movie {
        coverUrl
        averageRating
        genres { name }
      }
      ... on TVShow {
        coverUrl
        averageRating
      }
      ... on Game {
        coverUrl
        genre
        themes
      }
      ... on MusicAlbum {
        coverUrl
        label
      }
      ... on Book {
        coverUrl
        isbn
      }
    }
  }
}
```

<Note>
  The `relatedMedia` field may be empty immediately after adding a new item. The **Recently Viewed** tab shows an enrichment indicator on the cover while the background search is running. Once it completes, the detail screen will display related titles.
</Note>

## How ratings and activity affect recommendations

The recommendation engine weighs several signals from your `UserActivity` records:

| Signal | Effect |
|--------|--------|
| High rating (4–5 stars) | Increases traversal weight from that item to its neighbors |
| `Completed` status | Treated as a strong positive signal |
| `In Progress` status | Moderate signal; indicates active interest |
| `Want to Watch` status | Low signal; marks intent without confirmed interest |
| `Dropped` status | Negative signal; reduces weight of similar titles |
| No rating | Neutral; activity is counted but not weighted heavily |

Ratings are stored via the `rateMedia` mutation and linked to both the `User` and `Media` nodes. The `averageRating` field on each media item aggregates scores from all users who have rated it.

```graphql
mutation {
  rateMedia(mediaId: "some-uuid", score: 4.5) {
    score
    ratedAt
  }
}
```

## Triggering recommendations

Recommendations are computed by the backend based on your current graph state. To get fresh results:

<Steps>
  <Step title="Add or sync media">
    The more items you have logged, the more edges the engine can traverse. Add titles via the **Add New Media** tab, or sync a connected service from **Settings → Integrations**.
  </Step>
  <Step title="Rate and review what you have consumed">
    Open a media detail screen from the **Recently Viewed** tab and add a star rating or status update. Higher-rated items carry more traversal weight.
  </Step>
  <Step title="Pull to refresh on the home screen">
    Pull down on the **Recommended** tab to re-run `getRecommendations` and display updated results.
  </Step>
</Steps>

## Tips for improving recommendation quality

<Note>
  If `getRecommendations` returns an empty list, you have not yet synced enough data or logged enough activity. Add a few titles with ratings or completed statuses, then pull to refresh on the home screen.
</Note>

<AccordionGroup>
  <Accordion title="Connect multiple services in the same category">
    The richer your graph, the better the engine can reason about preferences. Connecting both Spotify and Apple Music, for example, gives the engine more music edges to traverse — which produces more accurate music recommendations and cross-media connections (e.g., a game's soundtrack linking it back to the game).

    See [Connecting Services](/guides/connecting-services) for setup steps.
  </Accordion>

  <Accordion title="Rate everything you have already consumed">
    Every rating creates a weighted edge in your preference graph. Even items you added a long time ago can be improved by going back and setting a star rating. Open any cover in the **Recently Viewed** tab and update the rating on the detail screen.
  </Accordion>

  <Accordion title="Use the Dropped status deliberately">
    When you mark something as **Dropped**, the engine treats it as a negative signal and reduces the weight of similar titles. This is the fastest way to push a genre, franchise, or creator you dislike out of your recommendation queue.
  </Accordion>

  <Accordion title="Sync connected services regularly">
    NQ only surfaces media that has been imported or manually added. Syncing weekly (or after a session of heavy consumption) keeps your activity data current. Go to **Account Settings → Integrations** and tap **Sync** next to any service.
  </Accordion>

  <Accordion title="Pick autocomplete suggestions when adding media">
    Selecting a result from the autocomplete dropdown on the Add tab attaches the correct external ID to the item (TMDB for movies and TV, IGDB for games, ISBN for books). This gives the backend enough context to fetch richer metadata and populate a more complete `relatedMedia` graph, which directly broadens the pool of available recommendations.
  </Accordion>

  <Accordion title="Wait for enrichment to complete after adding new items">
    After you add a title, the backend runs a recursive search to populate its `relatedMedia` links. Recommendations that pass through that item won't appear until enrichment completes. The **Recently Viewed** tab shows an indicator on the cover while the job runs — typically a few seconds to a minute depending on graph depth.
  </Accordion>
</AccordionGroup>

## Related queries

To explore the full recommendation data model, the following queries are also useful:

```graphql
# View your complete activity history with statuses and ratings
query {
  me {
    activities {
      id
      status {
        name
      }
      rating
      review
      startedAt
      finishedAt
      media {
        id
        title
      }
    }
  }
}
```

```graphql
# View ratings you have submitted
query {
  me {
    ratings {
      score
      ratedAt
      media {
        id
        title
      }
    }
  }
}
```

```graphql
# Check the enrichment status of a recently added item
query ($mediaId: UUID!) {
  recursiveSearchStatus(mediaId: $mediaId) {
    state      # IDLE | RUNNING | COMPLETED
    completedAt
  }
}
```
